# 《我用 AI 写了一个 Windows 限速工具》
## 教学视频剧本·简化版（面向非专业人士）

**预估时长：约 12 分钟**

---

> **[口播]** = 主讲人台词  
> **[动画]** = 动态示意图  
> **[画面]** = 截图或录屏  

---

## 开场：一个真实的烦恼

**[动画：卡顿的视频通话画面，对面的人像素化，右下角弹出通知「OneDrive 正在同步 8.3GB」]**

**口播：**

"某天下午，我在视频会议里对着客户讲方案，画面突然卡成了 PPT。打开任务管理器一看，OneDrive 后台同步正在以每秒 20MB 的速度大口吃带宽。我知道是它干的，但我没办法告诉它'你最多只能用 10%'。

这件事让我想做一个工具：能看到每个程序在用多少网速，然后能给它设一个上限。"

**[动画：任务管理器截图，圈出 OneDrive，旁边打出大字「知道是你，但管不了你」]**

**口播：**

"但这不是一个能用 Python 写个脚本解决的问题。它涉及 Windows 最底层的内核机制。换句话说，这是一个真正硬核的系统编程项目。

这时候我做了一个决定——让 AI 来帮我。"

---

## 第一幕：和 AI 的第一次对话

**[画面：Copilot 聊天界面，输入框里打出「我想在驱动层实现对进程的网络限速，你规划一下怎么实现吧。」]**

**口播：**

"我把需求发给了 GitHub Copilot。注意我问的不是'帮我写代码'，而是'帮我规划'。这一步很关键——你不能把一个模糊的想法直接扔给 AI，然后期待它变出完美的代码。你先要跟 AI 讨论方案，让它告诉你这件事有多复杂。"

**[动画：AI 回复的架构图逐步浮现：ETW 监控层 → 令牌桶 → WFP 过滤层，三层架构]**

**口播：**

"AI 给出了一个分三层的方案：第一层，用 Windows 内置的 ETW 机制监测每个程序的网络流量；第二层，用令牌桶算法控制允许通过的流量配额；第三层，用 WFP——Windows 过滤平台——来真正阻断超额的数据包。

我当时看到这个回答，心里只有两个字：听不懂。但没关系，这正是 AI 最好用的地方——你可以继续问。"

## 第二幕：AI 是老师，也是搭档

**[动画：像聊天记录一样，一问一答的气泡逐个出现]**

**口播：**

"接下来一周，我和 AI 反复对话。我问它'ETW 是什么'，它用打比方的方式解释：就像给 Windows 内核装了一个「行车记录仪」，每次网络收发数据都留一条记录，记录里有是哪个程序发的、发了多少字节。

我问它'令牌桶算法是什么'，它画了个动画：把允许通过的流量想象成水桶里的水，水龙头以固定速度往里滴水，数据包到来时要消耗水，水不够就等着。滴得慢，速度就慢。这就是限速的原理。"

**[动画：令牌桶示意图——水龙头、水桶、出水口，速度可调]**

**口播：**

"最神奇的是，AI 不只是解释，它还帮我写代码。我说'现在帮我实现令牌桶'，它直接给出了完整的 C++ 实现，还主动提醒我：'你应该用测试驱动开发，先写测试再写功能，这样以后改代码不容易出 bug。'

我照做了。AI 帮我写出了 10 个单元测试，每个测试对应令牌桶的一种行为，比如'令牌满了不能超过上限'，'消费失败时令牌数量不变'。我运行测试，看着终端里一行行变绿，有一种奇特的满足感——明明是 AI 写的代码，但跑通的那一刻我也跟着开心。"

**[画面：终端截图，10 个测试通过，绿色文字「PASSED」]**

## 第三幕：AI 也会犯错

**[动画：报错界面，红色错误信息刷屏]**

**口播：**

"项目进展到写内核驱动时，事情开始不对劲了。"

"内核驱动是 Windows 系统最深处的代码——它不运行在普通的用户空间，而是和操作系统内核共享同一块内存。一个 bug 不是程序崩溃，而是整台电脑蓝屏。AI 在这个阶段犯了好几个错。"

**[动画：聊天气泡里 AI 给出代码，编译报错，人工标红]**

**口播：**

"第一个坑：AI 写共享头文件时，用的是 C++ 风格的 `struct` 语法。但内核驱动是用 C 语言编译的，C 里 `struct` 必须加 `typedef`，不然编译器认不出类型名。AI 不了解这个区别，我花了半小时才定位到这个问题。"

**[动画：放大对比两段代码——错误的 C++ struct 和正确的 typedef struct]**

**口播：**

"第二个坑更隐蔽。内核驱动编译时需要在开头定义一个特殊的宏 `NDIS630`，如果不加，一堆底层类型会直接'消失'，编译报告几十个'未定义的标识符'。AI 写的代码里没有这一行，我查了快一个小时的资料才找到原因。

第三个坑是逻辑 bug：AI 对入站数据包加了一个'快速放行'的条件，用意是优化性能，但这个条件太宽泛，结果导致下行限速完全失效——规则看起来设置成功了，但流量该多快还是多快。"

**[动画：「设置了限速」但速率不变的录屏，然后找到并删除那几行代码，速率下降]**

**口播：**

"这三个坑让我意识到一件事：AI 是一个知识面极广但缺乏这类工程经验的搭档。它知道 ETW 是什么、WFP 是什么，但它不知道这条生产线上具体有哪些隐形的规矩。这些规矩只有踩过坑的工程师才知道。

所以正确的姿势是：让 AI 负责写出框架和大量的样板代码，节省 80% 的时间；但每一个涉及'规则'的细节，你自己要去核实。"

## 第四幕：最终跑通的那一刻

**[画面：PowerShell 终端，以管理员身份运行签名脚本，驱动加载成功「STATE: 4 RUNNING」]**

**口播：**

"内核驱动要加载进系统，必须有微软认可的数字签名。开发测试阶段，我们可以用 Windows 的'测试签名模式'，用自己生成的证书签名。这个过程大约要重启电脑、运行三个 PowerShell 脚本、在 Windows 服务里注册驱动——这一套流程，AI 帮我把每一步都打包进了脚本，我只需要双击运行。"

**[画面：NetMonitor 界面启动，状态栏显示「Driver: 已连接 | ETW: 运行中」]**

**口播：**

"应用启动，状态栏亮了。ListView 里开始出现各个程序的实时网速数据。"

**[屏幕录制：后台开始下载大文件，curl.exe 出现在列表，下行速率 15MB/s；选中它，点「设置下行限速」，输入 200，确定；速率在几秒内降到 200KB/s]**

**口播：**

"我在后台运行一个下载任务，速率 15MB 每秒。然后在 NetMonitor 里选中它，设置下行限速 200KB/s——点确定。

三秒后，速率降下来了。"

**[动画：整个项目的技术图谱，所有技术名词之间连线，呈现出复杂的网络]**

**口播：**
