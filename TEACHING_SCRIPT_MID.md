## 开场：一个让你抓狂的场景

**[画面：任务管理器，某个进程的网络一栏占满，其他进程卡死]**

**口播：**

"你有没有遇到过这个情况：正在开视频会，OneDrive 突然开始同步，把带宽全吃光了。任务管理器看得到，但你限不了它——Windows 没有内置的「限速这个进程」功能。

这个视频讲的是我怎么从零设计并实现了一个进程网速限制工具。从需求分析、架构设计、分层开发到最终部署——完整的开发过程。核心逻辑跑在 Windows 内核里，在这之前我从来没写过内核代码。AI 帮了我很大的忙，也坑了我三次。"

---

## 第一章：需求拆解——这个问题的本质是什么

**[动画：需求拆成三个模块：「看到流量」→「判断超限」→「执行限速」]**

**口播：**

"做任何工具，第一步是拆需求。限速拆成三步：看到流量、判断超限、执行限速。对应三个模块：监控用 ETW（Windows 内置的内核事件追踪），判断用令牌桶算法，执行用 WFP。

关键设计决策：执行逻辑放用户态还是内核？"

---

## 第二章：用户态方案——原型验证

**[动画：时间轴——用户态程序每 100ms 查流量 → 超限就加 Block 规则 → 100ms 后删掉]**

**口播：**

"我先做了个用户态原型：后台线程每 100ms 检查流量，超限就加封锁规则，100ms 后删掉。这个原型验证了整条链路能跑通，但暴露了性能问题：每次决策要在用户态绕一圈再回内核，高速流量下撑不住。

于是做了第一个重大设计变更：判断和执行都搬进内核，用户态只负责界面和下发规则。"

---

## 第三章：整体架构——三层分离

**[动画：架构图——上层「GUI + ETW 监控」，中间「IOCTL 通信协议」，下层「内核 WFP Callout + 令牌桶」]**

**口播：**

"最终架构分三层。上层：Win32 界面 + ETW 监控，显示进程列表和实时流量。中间是 IOCTL 通信协议——用户设置限速后，App 通过 `DeviceIoControl` 把 `{进程ID, 方向, 字节数/秒}` 传进内核。下层是 WFP Callout 驱动：每个网络包经过时 Windows 会调用我的回调函数，我查令牌桶表，放行或丢包。

分这么细的好处：每层可以独立开发、独立测试，不碰内核也能测大部分逻辑。"

---

## 第四章：和 AI 规划——框架对，细节要人审

**[画面：Copilot 对话截图，展示架构规划问答]**

**口播：**

"架构想好之后，我让 AI 出详细设计。它给了完整的模块划分和代码框架，和我的思路基本一致。但有个关键细节它漏了。

内核里要把「网络连接」和「进程」关联起来，用的 API 叫 `FwpsFlowAssociateContext0`——它有个硬性前置条件：目标 callout 必须注册 `flowDeleteFn` 清理回调，否则直接返回错误。这种约束藏在文档犄角旮旯里，AI 没覆盖到。

这定了合作模式：AI 出框架，我逐条审假设。"

---

## 第五章：开发策略——先在用户态把算法测透

**[代码：TokenBucket 的测试用例滚动展示]**

**口播：**

"这是整个项目最重要的开发策略：不急着写内核代码。

内核调试循环很慢——改代码、编译、签名、停服务、换文件、启服务，一轮三分钟。而令牌桶算法、滑动窗口统计、IOCTL 协议的序列化——这些都是纯逻辑，完全可以在用户态用 GoogleTest 跑。

AI 先写 10 个令牌桶测试用例，再写实现，还加了时间注入接口："

**[代码：两行接口声明]**

```cpp
void setTimeForTest(TimePoint tp);
void advanceTimeForTest(std::chrono::milliseconds ms);
```

**口播：**

"测试时直接跳到「1 秒后」验证，不用真 sleep。10 个令牌桶测试加 9 个滑动窗口测试，毫秒级跑完。

这样做的好处是：算法在用户态验证干净后，移植进内核只需要把 `std::chrono` 换成 `KeQueryPerformanceCounter`，把 `std::mutex` 换成 `KSPIN_LOCK`，核心逻辑一行没改。用户态的测试套件就成了内核代码的安全网。"

## 第六章：进内核——AI 踩的三个坑

**[动画：三颗地雷图标，依次亮起]**

**口播：**

"进了内核，AI 的代码出了三个问题。每个单独看都不大，但不了解背景会查很久。"

**[代码：并排展示 C++ struct 和 typedef struct]**

**口播：**

"**第一个坑：C/C++ struct 差异。**

驱动源文件是 `.c`，AI 写的共享头文件用了 C++ 风格的 struct。C 编译器不认无 typedef 的类型名，报一堆「未声明标识符」。改成 `typedef struct` 就好。不知道这个差异的话会以为是 include 路径问题，白查半天。"

**[代码：build-driver-manual.ps1 里加 NDIS630=1 的那行]**

**口播：**

"**第二个坑：神秘的版本宏。**

内核头文件里很多类型被 `NDIS_SUPPORT_NDIS630` 宏包裹。不加 `NDIS630=1`，报几十个「未声明标识符」，完全看不出跟版本号有什么关系。AI 的编译参数里没有这行。"

**[代码：高亮被删掉的那几行 if 逻辑]**

**口播：**

"**第三个坑：一段「好心」的放行逻辑。**

AI 加了一段判断：如果包带了「需要系统重新处理」的标记，直接放行。问题是 TCP 数据流里几乎所有入站包都带这个标记——结果怎么限速都不生效，而且没有任何错误日志。

靠在函数各出口加计数日志定位：99% 包走了 PERMIT 分支。删掉，解决。这类 bug 的根源：AI 的经验来自文档，不来自运行时。"

## 第七章：驱动签名与部署

**[动画：.sys 文件 → 签名检验 → 加载 或 报错 577]**

**口播：**

"编译过了，`sc start` 报 577：签名不合规。内核驱动必须签名才能加载。开发期用测试签名模式：`bcdedit /set testsigning on` + 重启，然后生成自签名证书，用 `signtool` 签名 .sys。AI 把整套打包成了一键脚本。注意：testsigning 重启后才生效。"

**[画面：命令行输出 STATE: 4 RUNNING]**

**口播：**

"重启后驱动加载成功。整条链路跑通：App 用 `CreateFileW` 打开驱动设备，`DeviceIoControl` 下发限速规则，内核收到后更新令牌桶表，之后每个数据包在 classify 回调里被逐一裁决。

从原型到内核驱动，开发顺序是：用户态原型验证链路 → TDD 稳住算法 → 内核实现 → 签名部署。每一步都建立在上一步的验证之上。"

**[屏幕录制：下载进程限速前后对比，限速后稳定在目标带宽]**

**口播：**

"效果达到了预期——限速后数秒收敛，稳定在目标带宽附近。

回顾整个开发过程，有三点值得提炼：

第一，**需求拆解决定了架构**。把「限速」拆成监控、判断、执行三个独立模块，才能分层开发、分层测试。

第三，**TDD 是跨层开发的安全网**。29 个测试保证了令牌桶和统计模块的正确性，移植进内核时不用担心算法层面的退化。

AI 在框架设计和样板代码上帮了大忙，三个内核 bug 都属于「运行时上下文假设不成立」——这类问题目前还需要人来兜底。"
