# NetMonitor 是怎样限速节省流量的？

## 引言

我们日常使用电脑时，常常遇到这样的烦恼：某个程序在后台疯狂下载，把整条网络带宽占满，导致其他应用卡顿。更令人头疼的是，如果你使用的是按流量计费的网络（比如手机热点、卫星网络、或者有月度流量上限的宽带），这些"流量大户"会迅速耗尽你的配额。

NetMonitor 就是为了解决这个问题而设计的。它能够监控每一个进程的网络流量，并对指定进程进行精确限速。限速的直接效果是降低了速度，但间接效果是：**在相同的使用时长内，消耗的总流量更少了。**

那么，软件层面的限速到底是怎么做到的？数据包在网络中传输时是"全速"的，一个软件怎么能让它"慢下来"？

答案是：**不是让包变慢，而是控制包的放行节奏。**

---

### 各模块职责

| 模块 | 职责 | 关键细节 |
|------|------|----------|
| **EtwMonitor** | 流量采集 | 启动 Windows 内核级 ETW 会话（NT Kernel Logger），监听 TCP/UDP 收发事件，将 (PID, 字节数, 方向) 回调给 TrafficTracker |
| **TrafficTracker** | 流量统计与速率计算 | 按进程聚合流量，每秒生成历史快照，支持滑动窗口查询（如"过去 10 分钟 PID 1234 下行了多少字节？"） |
| **AlertManager** | 报警策略引擎 | 管理多条报警策略，每秒评估一次。当某进程在指定时间窗口内的流量超过阈值时，自动触发限速 |
| **DivertLimiter** | 限速执行 | 本文的主角。使用 WinDivert 拦截数据包，通过令牌桶+排队机制实现精确限速 |
| **MainWindow** | 用户界面 | Win32 原生窗口，ListView 展示进程列表，右键/工具栏操作限速和报警，定时器驱动刷新 |

### 数据流向

```
网卡 → ETW 内核事件 → EtwMonitor → TrafficTracker (统计)
                                          ↓
                                    AlertManager (评估)
                                          ↓ 超阈值
                                    DivertLimiter (限速)

网卡 ←→ WinDivert 驱动 ←→ DivertLimiter (拦截/排队/放行)
                                          ↑
                                   用户通过 GUI 手动设置
```

可以看到，EtwMonitor 和 DivertLimiter 都与网卡打交道，但角色不同：**EtwMonitor 只"看"不"动"**（被动采集统计），**DivertLimiter 既"看"又"动"**（主动拦截和控制数据包）。

了解了整体架构，接下来我们深入限速的核心——DivertLimiter 是怎样一步步控制数据包的。

## 第一步：拦截——在网卡和应用之间设卡

NetMonitor 使用了一个叫做 WinDivert 的底层组件。WinDivert 的工作方式类似于高速公路上的收费站——它在 Windows 网络协议栈中插入了一个"检查点"，所有进出系统的 TCP 和 UDP 数据包都必须经过这个检查点。

```
                     ┌──────────────────────┐
  互联网 ──→ 网卡 ──→│  WinDivert 检查点     │──→ 应用程序
                     │  在这里决定：          │
                     │  ① 立即放行？          │
                     │  ② 暂存排队？          │
                     │  ③ 丢弃？             │
                     └──────────────────────┘
```

当一个数据包到达检查点时，WinDivert 会把它"扣留"——这个包不会被送到应用程序，而是交给 NetMonitor 来决定这个包的命运。NetMonitor 做完决策后，再把需要放行的包重新注入网络协议栈，送达应用程序。

上行（发送）方向也是同理：应用发出的包先被拦截，由 NetMonitor 决定是否立即放行。

---

## 第二步：识别——这个包属于哪个进程？

拦截到了包，但包本身只包含 IP 地址和端口号，并不标注"我是哪个进程发的"。那怎么知道该对谁限速呢？

NetMonitor 维护了一张**连接映射表（FlowTracker）**，记录着每条网络连接属于哪个进程：

```
连接映射表示例：

  协议    本地地址:端口        远端地址:端口        → 进程
  TCP    192.168.1.5:52301   93.184.216.34:443   → Chrome (PID 1234)
  TCP    192.168.1.5:52302   140.82.121.4:443    → Git (PID 5678)
  UDP    192.168.1.5:50000   8.8.8.8:53          → svchost (PID 900)
```

这张表通过两种方式来构建：

1. **实时监听**：WinDivert 的 Flow 层会在每个连接建立和断开时发出通知，通知中包含了进程 ID。NetMonitor 有一个专门的线程持续监听这些通知。

2. **查询系统表**：对于在 NetMonitor 启动之前就已经存在的连接，会查询操作系统的 TCP/UDP 连接表来补全映射关系。

有了这张表，当拦截到一个数据包时，只需要用包中的五元组（协议 + 源地址 + 源端口 + 目标地址 + 目标端口）去查表，就能知道它属于哪个进程。

## 第三步：决策——令牌桶算法

知道了包属于哪个进程之后，就要决定这个包能不能放行。这里用到了经典的**令牌桶（Token Bucket）**算法。

### 什么是令牌桶？

想象你面前有一个桶，桶里装着"令牌"。每个令牌代表允许传输 1 个字节的数据。

- **持续补充**：系统以固定速率向桶里投放令牌。比如限速 100 KB/s，那么每秒投入 102,400 个令牌。
- **上限封顶**：桶有容量上限（称为"突发量"），满了之后新令牌溢出丢弃，不会无限积累。
- **消费令牌**：每来一个数据包，就要从桶里取出与包大小相等数量的令牌。取到了，放行；取不到，排队等待。

```
 令牌补充速率：100 KB/s
       │
       ▼
  ┌─────────┐
  │ ○ ○ ○ ○ │ ← 桶中令牌（当前可用额度）
  │ ○ ○ ○   │
  │ ○ ○     │    容量上限 = 10 KB（允许短暂突发）
  └────┬────┘
       │
       ▼
  来了一个 1500 字节的包
  → 桶里有足够令牌？
     是 → 消费 1500 个令牌，放行！
     否 → 令牌不够，排队等待。
```

**为什么用令牌桶而不是简单计数？** 因为令牌桶天然支持"突发"——如果一段时间没有流量，令牌会积累到上限，之后可以短暂地允许超过平均速率的传输。这让网络行为更自然，避免了僵硬的匀速限制导致的卡顿。

### 代码中的实现

NetMonitor 为每个被限速的进程维护两个独立的令牌桶——一个管上行，一个管下行。令牌桶的补充通过计算时间差来实现，而非真的启动定时器投放：

```
补充量 = 流逝时间（秒） × 速率（字节/秒）
当前令牌 = min(当前令牌 + 补充量, 桶容量上限)
```

每次调用 `tryConsume` 时先计算补充、再尝试消费，一步完成。这种"惰性补充"的设计既简洁又精确。

## 第四步：排队——包不会被丢弃，而是等待

这一步是整个限速机制中**最关键的设计决策**。

当令牌不足时，数据包不会被直接丢弃，而是被放入一个**等待队列**。这就像高速公路入口的匝道信号灯——车不会被赶走，而是排队依次放行。

```
  数据包到达
      │
      ▼
  ┌───────────────┐
  │ 查令牌桶      │
  │ 令牌够？       │
  └──┬─────┬──────┘
     │     │
    是     否
     │     │
     ▼     ▼
   放行   排队等待
          （暂存在内存中）
          ↓
        每 10ms 检查一次：
          令牌桶补充了新令牌吗？
             是 → 从队首取包，消费令牌，重新注入放行
             否 → 继续等待
```

一个专门的**"节奏控制线程"（Pacer Thread）**每隔约 10 毫秒执行一次排队包的释放操作。它会查看每个进程的等待队列，尝试用新补充的令牌去消费队首的包：

- 令牌够 → 取出这个包，重新注入协议栈
- 令牌不够 → 停止释放，等下一个 10ms 周期

这样做的好处是：

1. **数据不丢失**：排队的包最终都会被送达，只是被延迟了。
2. **TCP 不会重传**：因为包没有丢，TCP 协议不会触发重传机制，避免了重复传输带来的流量浪费。
3. **速率平滑**：10ms 的释放周期使得数据流看起来是相对匀速的，而不是一阵一阵的。

### 队列的安全阀

当然，队列不能无限增长。NetMonitor 为每个进程设置了队列上限：

- 最多 **2000 个包**
- 或者最多 **3 MB** 数据

超过这个限制，新到的包才会被丢弃。这是一个安全阀——正常限速情况下几乎不会触发（因为 TCP 的拥塞控制会自动降低发送速率来适应），只有极端流量洪峰时才起作用。

## 限速如何节省流量？

读到这里你可能会问：包只是被延迟了，最终还是送达了，总量没变，怎么能说"节省流量"呢？

关键在于：**限速改变了应用程序的行为。**

### 1. 流媒体自适应码率

视频平台（如 YouTube、Netflix、B站）使用自适应码率技术（ABR）。当播放器检测到可用带宽只有 500 KB/s 时，会主动切换到较低分辨率的视频流。一部 1 小时的电影：

- 不限速（1080p）：约 4.5 GB
- 限速 500 KB/s（480p）：约 0.7 GB

**同样的内容，流量减少了 84%。**

### 2. TCP 拥塞控制的反馈

TCP 协议内置了拥塞控制算法。当 NetMonitor 延迟了数据包的送达，TCP 的发送窗口会自动收缩，通知远端服务器"慢一点发"。这意味着：

- 服务器实际发送的数据量减少了
- 不是在你这端扔掉数据，而是从源头就减少了发送
- 没有浪费的重传流量

### 3. 时间窗口约束

如果你只在固定时间使用网络（比如每天用 8 小时），限速直接约束了这段时间内能传输的最大数据量：

```
无限速：100 Mbps × 8h = 360 GB
限速到 1 Mbps：1 Mbps × 8h = 3.6 GB
```

## 为什么选择"排队"而不是"丢包"？

前面提到，NetMonitor 在令牌不足时选择排队而非丢包。这个设计选择对于"节省流量"这个目标至关重要。

如果选择丢包策略会发生什么？

```
丢包策略（不好）：
  包到达 → 令牌不足 → 丢弃！
  → TCP 超时未收到 ACK
  → TCP 重传同一个包
  → 这个包的流量被消耗了两次甚至更多次
  → 流量不减反增
```

```
排队策略（NetMonitor 的做法）：
  包到达 → 令牌不足 → 排队等待
  → 10ms 后令牌补充 → 释放并送达
  → TCP 正常收到，发送 ACK
  → 无重传，零浪费
  → TCP 拥塞窗口逐步收缩，从源头减速
```

排队策略不仅避免了浪费，还通过 TCP 的正常反馈机制让远端服务器主动降速。这是一个正循环：限速 → 延迟增大 → TCP 降速 → 服务器发得更少 → 流量真正减少。

## TCP 与 UDP：限速效果的差异

前面的分析主要以 TCP 为例。但网络上还有大量的 UDP 流量（视频通话、直播、游戏、DNS 查询等）。TCP 和 UDP 在限速节流方面有本质区别，理解这一点对合理使用 NetMonitor 非常重要。

### TCP：排队 → 反馈 → 源头减速

TCP 自带拥塞控制机制。当 NetMonitor 通过排队延迟了包的送达，整个反馈链条如下：

```
服务器发送数据
    ↓
包被 NetMonitor 排队延迟
    ↓
应用程序延迟收到数据 → 延迟发送 ACK（确认）
    ↓
服务器感知到 RTT（往返时间）增大
    ↓
TCP 拥塞控制算法自动收缩发送窗口
    ↓
服务器从源头降低发送速率
    ↓
实际穿过互联网的数据量减少 → 流量真正节省
```

这是一个"正反馈回路"——你这边限速，服务器那边自动配合减速。**TCP 限速对节省流量非常有效。**

### UDP：没有反馈，服务器不会配合

UDP 是"发射后不管"的协议——没有确认、没有重传、没有拥塞控制。服务器发出的每个包，无论你收不收得到，它都不关心。

这就导致了一个关键问题：

```
场景：某应用以 10 MB/s 通过 UDP 下载数据，你限速到 500 KB/s

服务器持续以 10 MB/s 发送 UDP 包
    ↓
所有包穿过互联网，到达你的网卡        ← 流量已在此刻消耗
    ↓
WinDivert 拦截，交给 NetMonitor
    ↓
令牌桶每秒只允许 500 KB 通过
    ↓
剩余 9.5 MB/s 的包进入排队
    ↓
队列在几百毫秒内达到上限（2000 包 / 3 MB）
    ↓
后续的包 → Drop 丢弃
    ↓
但服务器不知道你丢了，继续以 10 MB/s 发送
    ↓
结果：你付了 10 MB/s 的流量费，却只用到了 500 KB/s
```

**UDP 下行限速不能节省流量**——数据包已经穿过了你的互联网连接到达网卡，在用户态丢弃只是阻止了应用接收，流量费已经产生。

### 上行方向：TCP 和 UDP 都有效

无论是 TCP 还是 UDP，上行限速都是有效的：

```
应用程序要发送数据
    ↓
包被 WinDivert 拦截（还未离开本机）
    ↓
NetMonitor 排队等待令牌
    ↓
按限定速率逐步放行
    ↓
数据以受控速率离开网卡 → 流量被真正限制
```

因为包在发出之前就被拦住了，还没有消耗任何网络流量。

### 例外：部分 UDP 应用仍然有效

虽然 UDP 本身没有拥塞控制，但很多现代应用在 UDP 之上实现了自己的速率调节：

| 协议/应用 | 限速是否有效 | 原因 |
|-----------|-------------|------|
| **QUIC（HTTP/3）** | 有效 | QUIC 虽然跑在 UDP 上，但内置了完整的拥塞控制算法，会像 TCP 一样对延迟和丢包做出反应 |
| **WebRTC（视频通话）** | 有效 | 有 REMB/TWCC 带宽估计机制，会根据实际接收速率自动降低视频码率 |
| **自适应流媒体** | 部分有效 | 即使底层用 UDP，播放器也会根据接收速率切换到更低的码率 |
| **DNS 查询** | 无意义 | 单个查询只有几十字节，限速没有实际价值 |
| **游戏数据/P2P** | 无效 | 纯 UDP 流量，服务器不会减速，下行限速只能丢包，不能省流量 |

### 限速节流效果一览

| 场景 | 能否节省流量 | 说明 |
|------|-------------|------|
| TCP 下行 | ✅ 有效 | 拥塞控制从源头减速，服务器发得更少 |
| TCP 上行 | ✅ 有效 | 包在发出前被拦住，不消耗带宽 |
| UDP 上行 | ✅ 有效 | 同上，包在发出前被控制 |
| UDP 下行 | ❌ 无效 | 包已到达网卡，丢弃也已消耗流量 |
| QUIC/WebRTC 下行 | ⚠️ 部分有效 | 取决于应用层是否有速率反馈机制 |

### 实际影响有多大？

对于大多数日常使用场景，TCP 流量占据绝对主导。网页浏览、文件下载、应用更新、大部分流媒体（包括 YouTube、Netflix、B站的主流模式）都使用 TCP 或基于 QUIC（有拥塞控制的 UDP）。纯 UDP 下行的场景主要集中在：实时游戏、某些 P2P 协议、局域网发现广播等——这些通常不是"流量大户"。

因此在实际使用中，**NetMonitor 的限速对节省流量是有效的**，UDP 下行的局限性对大多数用户影响不大。

## 总结

NetMonitor 由五个核心模块协同工作：EtwMonitor 负责采集流量数据，TrafficTracker 负责统计和速率计算，AlertManager 负责自动报警触发限速，DivertLimiter 负责执行限速，MainWindow 提供可视化操作界面。

其中，DivertLimiter 通过四层机制实现精确限速：

| 层次 | 机制 | 作用 |
|------|------|------|
| 拦截 | WinDivert 内核检查点 | 所有数据包必经的关卡 |
| 识别 | FlowTracker 连接映射 | 知道每个包属于哪个进程 |
| 决策 | TokenBucket 令牌桶 | 按设定速率精确控制放行节奏 |
| 缓冲 | PacketScheduler 排队 | 延迟而非丢弃，零浪费 |

限速对节省流量的效果取决于协议类型：**TCP 和上行 UDP 非常有效**（通过拥塞控制反馈或发前拦截从源头减少传输量），**纯 UDP 下行效果有限**（包已到达网卡，丢弃不省流量）。好在日常使用中 TCP 流量占据主导，这一局限性影响不大。

**限速不是把水管切断，而是把水龙头拧小。水流变细了，水费自然就少了。** 但要记住：对于 UDP 下行，水龙头在你家里拧小了，水表在外面已经转过了——真正省水需要从水厂（服务器）那头控制流量。
